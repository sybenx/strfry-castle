This is the strongest revision yet — the elevation model dissolves my earlier Guests objection in a way I didn't propose and probably should have: instead of protecting *events* (which requires storing event ids forever), you protect *authors*, and react-warding turns your existing "like it" gesture into silent curation. The no-event-ids invariant is a genuinely good architectural law, and I note with respect that two of my three previous flags were resolved by *deletion* rather than repair — kind-5 voids gone (no multi-source void semantics to get wrong), pardon-list sync gone (no snapshot needed; pardons live in the ledger, which is already durable). DECISIONS.md itself is excellent — the graveyard/waiting-room split with reasoning is exactly what keeps a project light over time.

But removing kind-5 voids exposed a new bug, and it's the one thing I'd block a merge on:

**Flag 1 — zombie bans (critical).** Reports are re-fetched every cycle and there's no intake watermark or dedupe specified. Sequence: you report a pubkey → banned → later you pardon via the UI → next cycle, intake re-reads the *same old 1984* (it still exists on relays; you can't unpublish it — that was the whole reason voids existed) → re-ban. Your pardon silently loses to a ghost within CYCLE_MINUTES. Same problem at raid cadence: re-enumeration of a banned domain re-bans a pubkey you individually pardoned, every raid, forever. The fix costs almost nothing because the ledger already stores source event ids: (a) intake skips any report whose event id already appears as a ledger source — each report bans exactly once, ever; (b) domain re-enumeration skips currently-pardoned pubkeys. Semantics fall out correctly: a pardon holds against everything that came before it, and a *new* report after the pardon re-bans, which is what a fresh judgment should do. This needs two sentences in Ban intake and two checklist lines ("pardoned pubkey stays pardoned across cycles despite the old report persisting; pardoned pubkey survives domain re-enumeration").

**Flag 2 — the state invariant reads as self-contradicting.** "Never event ids" is stated absolutely, but the ledger carries source event ids and follows.json stores one — correctly. A literal-minded session will either refuse to record provenance or "fix" the ledger. Reword the invariant to what you mean: *no event ids as retention or protection targets; provenance ids (source of a ban, source kind-3) are fine.* The standing order in PLAN.md has the same wording issue.

**Flag 3 — the chain of command vanished; was that intentional?** Last round added the constable/seneschal/bailiff/reeve offices and, more importantly, the enforcement rule that exactly one module may invoke `strfry delete`. This upload reverts to `gatekeeper` and a flat steward file list, with no trace in DECISIONS.md's graveyard — which its own rules say is where rejected ideas go so they aren't relitigated. If the offices were cut as theming overhead, fine and defensible — but the delete-confinement rule was a real safety property (dry-run, batching, and keep-conditions enforced in one choke point), and it's now nowhere. Minimum salvage: one standing order — "all `strfry delete` calls go through the single strfry-CLI wrapper; only raid.go and the purge step call it" — and a graveyard entry for the offices so a future session doesn't reinvent them.

**Flag 4 — manual raids default means unbounded courtyard growth.** "At the Lord's pleasure" is great theming and a defensible default, but the original complaint that started this whole project was Chronicle's resource weight on an Umbrel, and the new default converts retention from a schedule into a chore you can forget. Cheap mitigation, no policy change: reeve-side courtyard size in stats.json (event count is already there — add approximate bytes if `strfry scan --count` can be paired with DB size from `du` on... actually simplest: LMDB file size via the wrapper), and towncrier nudges when the Wild West crosses a threshold. The Lord can neglect his raids, but the town crier should be shouting about it.

Small notes, take or leave: react-warding resolves the reacted note's author "from the local relay" — if you reacted to a note that never landed in the castle, the ward silently fails; a fallback fetch of the e-tagged event from PUBLIC_RELAYS at resolution time is transient lookup, not stored state, so it doesn't violate the invariant. Archiving a *ward* via the scribe sends `{"authors":[ward]}` REQs to public relays — a mild metadata signal of the castle's interest in that pubkey; probably fine given ward privacy is declared obscurity-not-cryptography, but it belongs in the accepted-trade-offs list with the others. And towncrier's "everything public here is already public via CLI queries" claim is no longer quite true — the evicted list and grace expiries are admin state, published deliberately; reword to "published by choice" so nobody treats the claim as a privacy proof.

Verdict: fix Flag 1 before Phase 3a gets built, reword Flag 2 before any session reads it literally, make a conscious call on Flags 3 and 4. Everything else — scribe design (local-websocket replay instead of exec is lovely), NIP-05-as-membership, ledger versioning, the empirical ephemeral-kind verification in Phase 1, strict bytecheck — is ready as written.
